# dogAgent 系统架构

## 一句话介绍

dogAgent 是一个基于 RAG + 多层记忆的雪纳瑞犬知识助手，支持多用户多会话、对话压缩、知识库检索、偏好记忆和联网搜索。

## 一次对话的完整数据流

用户输入 "雪纳瑞胰腺炎怎么治" 后，系统经过以下步骤：

### Step 1: 查询重写
```
原始查询: "胰腺炎怎么治"
        ↓ QueryRewriter（有历史消息时调 LLM 补全上下文）
重写后:  "雪纳瑞犬的胰腺炎治疗方案"
```
**为什么需要这步？** 用户常说"它怎么办"，没有主语。重写器结合上下文把代词补全，让检索更精准。

### Step 2: 知识库检索（两层 LLM 索引）
```
LLM 读 wiki/index.md（847字节，精简分类目录）
  → 判断: "这个问题属于 03-健康医疗"
LLM 读 wiki/03-健康医疗/index.md（3.4KB，每篇一行摘要）
  → 选中: ["犬胰腺炎", "胰腺炎饮食管理"]
加载这两篇文章的完整内容
```

### Step 3: 记忆上下文组装
```
从 context_items 取:
  - summary 条目 → XML 格式注入 system prompt
  - message 条目 → 作为对话历史（fresh tail）

从 user_preferences 取:
  - "用户养了一只迷你雪纳瑞叫旺旺" → 注入 system prompt
```

### Step 4: LLM 调用（带 tool calling）
```
[SystemMessage]: 系统提示 + 记忆上下文（摘要 + 偏好）
[HumanMessage]: "你好" (历史)
[AIMessage]: "你好！" (历史)
[HumanMessage]: 知识库内容 + 用户问题

LLM 回答...
  工具 1: memory_expand(summary_id) → 展开历史摘要回溯原始消息
  工具 2: web_search(query)         → 知识库不足时搜索互联网补充
  → LLM 综合工具结果继续回答（最多 3 轮 tool calling）
```

### Step 5: 持久化 + 后台任务
```
保存: conv_store.add_message()  ← 同时写入 conversations 表 + context_items
触发后台（不阻塞用户）:
  1. 偏好提取: LLM 从对话中提取用户信息
  2. 压缩检查: 如果 context token > 6000，触发 DAG 压缩
  3. 标题生成: 第 3 轮对话后自动生成会话标题
```

## 核心组件说明

### API 层 (`api/server.py`)

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/auth/register` | POST | 注册，返回 JWT token |
| `/api/auth/login` | POST | 登录，返回 JWT token |
| `/api/auth/me` | GET | 获取当前用户信息 |
| `/api/chat` | POST | 核心对话接口（检索+记忆+LLM+联网搜索） |
| `/api/sessions` | GET/POST | 会话列表/新建会话 |
| `/api/sessions/{id}` | DELETE | 删除（归档）会话 |
| `/api/sessions/{id}/messages` | GET | 获取会话消息（从 context_items 视图） |
| `/api/preferences` | GET/POST | 用户偏好查看/添加 |

### Chat Engine (`agent/chat.py`)

chat.py 是整个系统的**粘合层**，把检索、记忆、LLM 串在一起：

```python
# 核心函数调用链
rewritten_query = query_rewriter.rewrite(user_input, recent_msgs)
results = retriever.retrieve(rewritten_query, top_k=3)
wiki_context = retriever.format_context(results)
memory_context = build_memory_context(memory, user_id, session_id)
tail_msgs = summary_dag.get_context_messages(user_id, session_id)  # 从 context_items 取
messages = build_messages(SYSTEM_PROMPT, memory_context, wiki_context, tail_msgs, user_input)
answer = invoke_with_tools(llm, messages, [memory_expand_tool, web_search_tool])
```

### 多租户隔离

所有数据表都有 `user_id + session_id` 联合索引：
- **对话和摘要**: 严格按 session 隔离
- **偏好**: 按 user_id 跨 session 共享（"我家狗叫旺旺" 在所有会话中都能用）
- **认证**: JWT token 包含 user_id，每个请求校验

## 遇到的问题和解决方案

### 问题 1: server.py 还在用已删除的 `create_embedder`

**现象**: server.py 的 `_init_components` 调用了 `create_embedder()` 和 `create_memory_system(embedder=embedder)`，但这两个函数在清理 MemoryIndex 时已经删除。

**解决**: 直接改为 `create_memory_system()`，去掉 embedder 参数。

### 问题 2: Web 端 tail_msgs 和 CLI 不一致

**现象**: chat.py 已经改用 `get_context_messages()` 从 context_items 取 fresh tail，但 server.py 还在用 `recent_msgs[-6:]`。压缩后 Web 端会看到已经被压缩掉的旧消息。

**解决**: server.py 第 284 行也改为 `dag.get_context_messages(user_id, session_id)`，和 chat.py 保持一致。context_items 是唯一数据源。

### 问题 3: 记忆功能在浏览器中不工作

**现象**: 用户问 "我是谁"，系统回答 "我无法识别你的身份"。

**根因**: SYSTEM_PROMPT 只说 "基于知识库回答"，没有告诉 LLM 使用记忆中的用户信息。

**解决**: 重写 SYSTEM_PROMPT，明确区分两类信息源：
- 记忆（偏好/摘要）→ 直接使用，用于回答个人问题
- 知识库（Wiki）→ 引用来源，用于回答专业问题

### 问题 4: `/api/sessions/{id}/messages` 返回的是原始消息，不是 context_items 视图

**现象**: 用户切换会话后看到的是所有原始消息，而 LLM 看到的是压缩后的 context_items（摘要 + fresh tail）。数据口径不一致。

**解决**: 改为从 context_items 读取，summary 条目返回 `role="summary"`，message 条目返回原始消息。前端渲染 summary 为蓝色左边框的摘要卡片。

### 问题 5: 会话没有删除机制

**现象**: 侧栏只能新建和切换会话，堆积越来越多无用会话，用户体验差。

**解决**: 
- 后端加 `DELETE /api/sessions/{id}` 端点，调用 `SessionManager.archive_session()`（软删除，设 `is_active=0`）
- 前端会话列表悬停时显示 × 删除按钮，点击确认后删除并刷新列表
- 软删除而非硬删除，数据可恢复

### 问题 6: 无法回答知识库外的问题

**现象**: 用户问 "帮我找雪纳瑞训练教程"，系统只能复述知识库内容，无法给出实际的外部链接或最新信息。

**解决**: 集成 `web_search` tool（基于 DuckDuckGo HTML 搜索，无需 API key）：
- 在 chat.py 中创建 `create_web_search_tool()`，注册为 LangChain tool
- 和 `memory_expand` 一起传给 `invoke_with_tools()`，LLM 自主决定何时调用
- 更新 SYSTEM_PROMPT，明确告诉 LLM 有三类信息源（记忆/知识库/网页搜索）
- LLM 判断知识库不足时自动搜索互联网，将结果整合到回答中

## 面试怎么讲

> "dogAgent 是一个基于 RAG 的知识问答系统。和普通 RAG 不同的是，它有四个核心创新：
> 1. **两层 LLM 索引检索** — 不用向量数据库，而是让 LLM 读精简的分类索引来判断相关文章，类似 Karpathy 的 LLM Wiki 思路
> 2. **Lossless Context Management** — 用 DAG 结构压缩对话历史，但任何信息都不会真正丢失，LLM 可以通过 tool calling 随时展开摘要回溯到原始消息
> 3. **context_items 统一视图** — 摘要和消息在同一个有序表中管理，LLM 的 system prompt 和对话历史都从这张表读取，保证数据一致性
> 4. **Tool Calling 扩展能力** — LLM 不只是问答，它有 memory_expand（回溯记忆）和 web_search（联网搜索）两个工具，能自主判断何时需要更多信息并主动获取"
