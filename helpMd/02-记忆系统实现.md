# dogAgent 记忆系统实现（LCM - Lossless Context Management）

## 核心理念

LLM 的上下文窗口有限（如 8K/128K token），但用户和 AI 的对话可能持续几百轮。传统方案是截断最早的消息，但这会导致 **信息永久丢失** — 用户三天前说过的 "我家狗叫旺旺" 被截掉后，AI 就不认识旺旺了。

LCM（Lossless Context Management）的解决思路：**压缩但不丢失**。用 DAG（有向无环图）把旧对话压缩成摘要，但原始消息始终保留在数据库中，LLM 可以随时通过 tool calling 展开摘要回溯到原文。

## 数据模型：5 张表

```
conversations        summaries           summary_edges
┌──────────────┐    ┌──────────────┐    ┌───────────────┐
│ id           │    │ id           │    │ parent_id     │
│ user_id      │    │ user_id      │    │ child_id      │
│ session_id   │    │ session_id   │    │ child_type    │
│ role         │    │ depth        │    └───────────────┘
│ content      │    │ content      │
│ token_count  │    │ token_count  │    context_items（核心!）
│ created_at   │    │ source_start │    ┌───────────────┐
└──────────────┘    │ source_end   │    │ user_id       │
                    └──────────────┘    │ session_id    │
                                        │ ordinal       │ ← 全局顺序
user_preferences                        │ item_type     │ ← 'message' | 'summary'
┌──────────────┐                        │ message_id    │ ← 指向 conversations
│ id           │                        │ summary_id    │ ← 指向 summaries
│ user_id      │                        └───────────────┘
│ content      │
│ superseded_by│ ← 级联更新链
│ last_confirmed_at │ ← 过期检测
└──────────────┘
```

### context_items：为什么它是核心？

**设计决策**：context_items 是 LLM 看到的对话上下文的 **唯一数据源**。

它是一个有序列表，`ordinal` 从 0 开始递增，每个条目要么是一条原始消息（`item_type='message'`），要么是一个摘要（`item_type='summary'`）。

LLM 的输入从这里构建：
- summary 条目 → XML 格式注入 system prompt（`<summary id="..." depth="0">内容</summary>`）
- message 条目 → 作为多轮对话历史（即 "fresh tail"）

**关键操作 `replace_context_range()`**：压缩时，把一段连续的 message 条目替换成一个 summary 条目，然后重排 ordinal 保持连续。

```
压缩前 context_items:
  ordinal 0: message "问题1"
  ordinal 1: message "回答1"
  ordinal 2: message "问题2"
  ordinal 3: message "回答2"
  ordinal 4: message "问题3"  ← fresh tail
  ordinal 5: message "回答3"  ← fresh tail

压缩后:
  ordinal 0: summary "用户讨论了问题1和问题2"  ← 替换了 0-3
  ordinal 1: message "问题3"                    ← 重排
  ordinal 2: message "回答3"                    ← 重排
```

## DAG 压缩引擎（CompactionEngine）

### 触发条件

```python
should_compact = context_token_count > context_budget * context_threshold
# 默认: 总 token > 8000 * 0.75 = 6000 时触发
```

### 两阶段压缩

#### Phase 1: Leaf 压缩（消息 → 摘要）

```
context_items 中的连续消息（排除 fresh tail）
  ↓ 选最旧的一段连续消息（_select_oldest_leaf_chunk）
  ↓ 发给 LLM 生成摘要（目标 600 token）
  ↓ replace_context_range() 原子替换
```

**选块算法严格遵循 LCM 规范**：
1. 跳过开头的非 message 条目（`continue`）
2. 一旦开始收集 message，遇到 summary 立即停止（`break`）
3. 不越过 fresh tail 边界

```python
# 伪代码
for item in context_items:
    if item.ordinal >= fresh_tail_ordinal:
        break  # 不碰 fresh tail
    if item.type != 'message':
        if collecting:
            break  # 遇到非 message 停止
        continue   # 跳过开头的 summary
    chunk.append(item)
    collecting = True
```

#### Phase 2: Condensation（摘要 → 更高层摘要）

当同一 depth 的连续 summary 数量 >= 6（min_fanout）时，合并为更高层摘要：

```
depth 0: [leaf1] [leaf2] [leaf3] [leaf4] [leaf5] [leaf6] [msg] [msg]
                          ↓ condensation
depth 1: [condensed_summary]                              [msg] [msg]
```

**选块算法相同**：跳过开头不同 depth 的 summary，收集连续同 depth 的，遇到不同 depth 停止。

### 循环直到满足条件

```python
for round in range(max_rounds):  # 最多 5 轮
    leaf_created = _compact_leaves()
    condensed_created = _condense_all_levels()
    if not should_compact():
        break  # token 已降到阈值以下
    if leaf_created == 0 and condensed_created == 0:
        break  # 没有可压缩的了
```

### Fresh Tail 保护

最后 8 条消息永远不被压缩。这保证 LLM 总能看到最近的对话内容，不会出现 "刚说完的话就被忘了" 的情况。

```python
def _resolve_fresh_tail_ordinal():
    # 从末尾往前数，找到第 fresh_tail_count 个 message 的 ordinal
    msg_count = 0
    for item in reversed(context_items):
        if item.type == 'message':
            msg_count += 1
            if msg_count >= fresh_tail_count:
                return item.ordinal
    return 0  # 消息不足，全部保护
```

## 摘要 Prompt 设计

### Leaf 摘要 Prompt

```
请将以下对话内容压缩为一段简洁的摘要。
要求：
1. 保留所有关键信息（诊断、建议、用药、数据等）
2. 保留时间线和因果关系
3. 目标长度约 600 个 token
4. 使用第三人称描述
5. 不要添加对话中没有的信息
6. 最后一行必须是：Expand for details about: <简短列举被省略的具体细节>
```

**"Expand for details about:" 这行是关键** — 它告诉 LLM 这个摘要省略了哪些细节。当用户后续问到这些细节时，LLM 会决定调用 `memory_expand` 工具来展开。

### 生成的摘要示例

```xml
<summary id="abc123" depth="0" kind="leaf" created="2024-01-15">
用户讨论了旺旺的耳道感染治疗。助手建议使用耳肤灵滴耳液（每天两次），
并口服阿莫西林克拉维酸钾（12.5mg/kg）。治疗周期约 7-10 天。
Expand for details about: 具体用药频率、剂量计算方式、复查时间建议
</summary>
```

## 无损回溯（Lossless Claw Drill-Down）

当 LLM 看到摘要中的 "Expand for details about: 具体用药频率..." 时，如果用户问 "旺旺的药具体怎么吃"，LLM 会调用 `memory_expand(summary_id)`：

```
memory_expand("abc123")
  ↓ MemoryDrillDown.drilldown_by_id()
  ↓ SummaryDAG.drill_down() → 查 summary_edges 表
  ↓ 子节点类型判断:
     - 如果是 summary → 返回子摘要内容 + id（LLM 可再次展开）
     - 如果是 message_range → 从 conversations 表取原始消息
  ↓ 返回给 LLM，LLM 用这些信息继续回答
```

**整个过程由 LLM 自主决定**：系统不决定什么时候展开，LLM 通过 tool calling 机制自己判断。最多 3 轮 tool calling。LLM 同时还可以调用 `web_search` 工具搜索互联网 — 两个 tool 并列注册，LLM 根据需要选择调用哪个。

## 用户偏好系统

### 偏好提取（后台自动）

每轮对话后，后台任务把对话内容发给 LLM 提取用户信息：

```
用户: "旺旺今天 3 岁了，是个公的迷你雪纳瑞"
  ↓ LLM 提取
{
  "new": ["用户的狗叫旺旺，3岁公犬，迷你雪纳瑞"],
  "update": []
}
```

### 级联更新

如果用户说 "旺旺今年 4 岁了"，LLM 会识别出这是对已有偏好的更新：

```python
# 旧偏好不删除，而是标记被取代
old_pref.superseded_by = new_pref.id
# 查询时只返回 superseded_by IS NULL 的
```

**为什么不直接删除？** 保留更新链可以追溯偏好的变化历史。

### 偏好过期检测

```python
STALE_DAYS = 30
def _is_stale(pref):
    return (now - pref.last_confirmed_at).days > STALE_DAYS
```

过期的偏好在注入 prompt 时会标注 "[可能已过时]"，提醒 LLM 谨慎使用。每次对话时 `touch_preferences()` 刷新确认时间。

## 会话管理

### 多会话隔离

- 每个用户可以有多个会话
- conversations、summaries、context_items 都按 `(user_id, session_id)` 隔离
- 偏好是**跨会话**的（只按 user_id 区分）

### 自动标题生成

```python
def needs_title(session_id):
    msg_count >= 6 and title is None
```

满足条件后，后台任务取前 6 条消息发给 LLM 生成一个 10 字以内的标题。

## 遇到的问题和解决方案

### 问题 1: get_top_summaries 旧逻辑导致摘要不可见

**现象**: 调用 `add_summary()` 后，`get_context_text()` 返回空字符串。

**根因**: 旧代码用 `MAX(depth)` 从 summaries 表直接取最高层摘要，但新设计中 context_items 才是数据源。如果没有把 summary 写入 context_items，`get_context_text()` 就找不到它。

**解决**: 删除 `get_top_summaries` 旧逻辑，添加 `append_context_summary()` 辅助方法。所有通过 compaction 创建的 summary 都会通过 `replace_context_range()` 自动写入 context_items。

### 问题 2: `replace_context_range()` ordinal 重排时 UNIQUE 冲突

**现象**: 删除中间段再 INSERT 新 summary + UPDATE 后续 ordinal 时，可能出现 ordinal 重复。

**解决**: 两遍重排 — 先把所有后续 ordinal 改为负数（临时值），再改为正确的连续值：

```python
# Step 1: 删除被替换的范围
DELETE WHERE ordinal BETWEEN start AND end
# Step 2: 插入新 summary
INSERT (ordinal=start_ord, type='summary', summary_id=xxx)
# Step 3: 重排（两遍避免冲突）
UPDATE SET ordinal = -(ordinal - gap + 1) WHERE ordinal > end  # 先改负
UPDATE SET ordinal = -ordinal WHERE ordinal < 0                  # 再改正
```

### 问题 3: 跨 session 记忆被移除后测试失败

**现象**: 删除 MemoryIndex（向量检索模块）后，test_e2e.py 中 `"memory_idx": self.memory_idx` 报 AttributeError。

**解决**: 从测试基类的 memory dict 中移除 `memory_idx` 字段。设计决策：只有 user_preferences 需要跨 session，对话历史和摘要严格按 session 隔离。

### 问题 4: context_items 和对话历史不一致

**现象**: chat.py 用 `get_context_messages()` 取 fresh tail，但 server.py 还用 `conv_store.get_recent()[-6:]`。压缩后两端看到的历史不同。

**解决**: server.py 也改用 `get_context_messages()`，确保 CLI 和 Web 端都从 context_items 这个唯一数据源读取。

## 关键参数配置

| 参数 | 默认值 | 含义 |
|------|--------|------|
| context_budget | 8000 token | 上下文总预算 |
| context_threshold | 0.75 | 触发压缩的阈值比例（6000 token） |
| fresh_tail_count | 8 | 保护最后 N 条消息不被压缩 |
| leaf_target_tokens | 600 | 叶子摘要目标大小 |
| condensed_target_tokens | 900 | 浓缩摘要目标大小 |
| min_fanout | 6 | 最少多少个同 depth 摘要才触发 condensation |
| max_rounds | 5 | 单次压缩最多几轮 |
| STALE_DAYS | 30 | 偏好过期天数 |
| max_drilldown_tokens | 2000 | 单次 drill-down 返回的最大 token |

## 面试怎么讲

> **问: 你的记忆系统怎么解决上下文窗口有限的问题？**
>
> "我们用 **DAG 压缩 + 无损回溯** 的方案。核心是一张 context_items 表，它维护了一个有序列表，里面混合了原始消息和摘要。当 token 超预算时，压缩引擎把最旧的连续消息压缩成一个 leaf summary，替换 context_items 中的对应段。如果 leaf summary 积累到 6 个以上，再合并成更高层的 condensed summary。
>
> 但信息不会丢失 — 每个摘要都记录了它来自哪些消息（source_start_id, source_end_id），通过 summary_edges 表形成 DAG。LLM 通过 tool calling 机制（memory_expand 工具）可以随时展开任一摘要，逐层回溯到原始消息。
>
> 这和普通的截断方案本质区别是：截断是**不可逆的信息丢失**，而 DAG 压缩是**可逆的信息压缩**。"

> **问: fresh tail 是什么？为什么要保护它？**
>
> "最后 8 条消息永远不被压缩。因为用户说话通常是连贯的 — 如果用户说了 '旺旺昨天拉肚子'，紧接着问 '需要吃什么药'，LLM 必须看到前一句才能回答。如果连最近的消息都被压缩了，LLM 就会失去对话的连贯性。8 条是我们实测的平衡点 — 足够覆盖最近 4 轮对话，同时不占太多 token。"

> **问: 偏好为什么跨 session 而对话历史不跨？**
>
> "偏好是关于用户的持久信息（'我家狗叫旺旺'），在所有会话中都应该被记住。而对话历史是特定话题的讨论（'今天聊了胰腺炎'），跨会话没有意义。这也是为什么偏好表只有 user_id 没有 session_id，而 conversations 和 summaries 都有 session_id。"
