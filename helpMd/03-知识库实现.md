# dogAgent 知识库实现

## 核心理念：LLM Wiki（Karpathy 风格）

传统 RAG：文档 → 向量化 → 向量数据库 → 相似度检索 → LLM 回答。

我们的方案：**LLM 自己维护一个结构化的 Wiki 知识库**。不用向量数据库，检索时让 LLM 读索引目录来判断哪些文章相关。

这个思路来自 Karpathy 的 LLM Wiki 概念：

> 不是每次查询都从原始文档重新检索，而是 LLM **持续构建和维护一个 Wiki** — 当新来源加入时，LLM 读取它、提取关键信息、整合进现有 Wiki。知识被编译一次然后保持更新，而不是每次查询时重新推导。

## 三层架构

```
Raw Sources (crawlers/)     →  不可变的原始数据
    ↓ clean_and_organize
Knowledge (knowledge/)      →  分类整理后的源文件
    ↓ build_wiki (LLM)
Wiki (wiki/)                →  LLM 生成的结构化 Wiki
    ↓ build_index
Index (index.json + index.md) → 检索用的两层索引
```

### 每层的职责

| 层 | 谁写 | 谁读 | 修改频率 |
|----|------|------|----------|
| Raw Sources | 爬虫 | build_wiki | 仅新增，不修改 |
| Knowledge | clean_and_organize | build_wiki | 新增 + 标记 wiki_processed |
| Wiki | build_wiki (LLM) | Retriever + 用户 | build_wiki 生成 + lint 维护 |
| Index | build_index | LLM Retriever | 每次 build_index 重建 |

## 知识库构建流水线

### Step 1: 爬虫（`crawlers/scripts/`）

6 个爬虫源：

| 来源 | 可靠性 | 内容 |
|------|--------|------|
| PetMD | 高 | 疾病百科、营养、护理（518 页已抓） |
| AKC | 高 | 品种标准、训练、繁殖 |
| VCA | 高 | 兽医院指南 |
| ASPCA | 高 | 有毒食物、中毒急救 |
| 波奇网 | 中 | 中文雪纳瑞信息 |
| Reddit | 中 | 社区经验 |

每个爬虫继承 `BaseCrawler`，实现：
- `extract_article_content()` — 从 HTML 提取正文
- `crawl()` — 爬取策略（直接 URL + 分类浏览 + 搜索关键词）

输出：`crawlers/raw_data/{source}/*.html` + `crawlers/cleaned_data/{source}/*.md`

### Step 2: 清洗整理（`clean_and_organize.py`）

**核心原则：零信息损失** — 只去掉确定的垃圾（script/style 标签），保留一切正文。

```
cleaned_data/petmd/*.md
    ↓ 读取 frontmatter 中的 category
    ↓ 映射到目标目录
knowledge/03-健康医疗/Pancreatitis-in-Dogs.md
```

分类映射：
```python
CATEGORY_DIR_MAP = {
    "品种百科": "01-品种百科",
    "饮食营养": "02-饮食营养",
    "健康医疗": "03-健康医疗",
    "美容护理": "04-美容护理",
    "训练与行为": "05-训练与行为",
    "日常饲养": "06-日常饲养",
    "繁殖与幼犬": "07-繁殖与幼犬",
    "法规与常识": "08-法规与养犬常识",
}
```

### Step 3: LLM 补缺（`generate_topics.py`）

爬虫抓不到的主题（如"手剥技术"、"中国养犬规定"），用 LLM 生成。

```
target_topics.json (52 个目标主题)
    ↓ scan_existing_knowledge() 扫描已有文件
    ↓ find_gaps() 找缺口
    ↓ generate_article() 调 LLM 生成
knowledge/04-美容护理/手剥技术详解.md
```

生成的文章标注 `source: LLM-generated` + `reliability: 低` + `needs_review: true`，和爬取来的内容区分。

### Step 4: Wiki 生成（`build_wiki.py`）

**核心创新：同义词分组**

PetMD 有一篇英文 "Pancreatitis in Dogs"，LLM 也生成了一篇中文 "犬胰腺炎"。它们应该合并为一个 Wiki 页面，而不是两个。

```python
TOPIC_SYNONYMS = {
    "pancreatitis": ["胰腺炎", "胰脏"],
    "bladder stones": ["膀胱结石", "urolithiasis", "尿路结石"],
    "hypothyroidism": ["甲状腺功能减退", "甲减"],
    "hand stripping": ["手剥", "手拔", "剥毛"],
    ...
}
```

`_normalize_topic_key()` 把标题映射到 canonical key：
- "Pancreatitis in Dogs" → `03-健康医疗::pancreatitis`
- "犬胰腺炎（Pancreatitis in Dogs）" → `03-健康医疗::pancreatitis`
- 同一 key 的文章归到同一组

多源合并时，LLM 会：
1. 整合不同来源的信息
2. 检测冲突（数据不一致时标注，不取舍）
3. 生成统一的中文 Wiki 页面

### Step 5: 索引构建（`build_index.py`）

生成**两层索引**：

#### 第 1 层：`wiki/index.md`（~1KB）

```markdown
# 雪纳瑞知识库索引

## 品种百科（3 篇）
迷你雪纳瑞、标准雪纳瑞、巨型雪纳瑞

## 健康医疗（47 篇）
犬胰腺炎、犬膀胱结石、犬耳部感染...等（共 47 篇）
→ 详见 03-健康医疗/index.md

## 美容护理（4 篇）
手剥技术、毛发护理、标准造型、耳部清洁
```

LLM 先读这个，判断去哪个分类。

#### 第 2 层：`wiki/{category}/index.md`

```markdown
# 健康医疗索引
- **犬胰腺炎** — 雪纳瑞高发，症状/诊断/治疗/低脂饮食方案
- **犬膀胱结石** — 草酸钙/鸟粪石结石，手术与饮食溶解
```

LLM 读分类索引，选具体文章。

## 两层 LLM 检索（LLMWikiIndexRetriever）

### 检索流程

```
用户: "雪纳瑞胰腺炎怎么治"
    ↓
Step 1: LLM 读 wiki/index.md (1KB)
    → ["03-健康医疗"]
    ↓
Step 2: LLM 读 wiki/03-健康医疗/index.md (3.4KB)
    → ["犬胰腺炎（Pancreatitis in Dogs）", "雪纳瑞犬的胰腺炎饮食管理"]
    ↓
Step 3: 加载这两篇文章的完整内容
    → 注入 LLM prompt 作为知识上下文
```

### 为什么不用向量数据库？

1. **语义理解更强** — LLM 能理解 "耳朵红红的" 应该找 "犬耳部感染"，向量相似度可能匹配不上
2. **零基础设施** — 不需要 embedding 模型、向量数据库、维护索引
3. **精简索引即可** — 85 篇文章的索引只有 1KB + 3.4KB，LLM 轻松处理
4. **Karpathy 验证过** — "在 ~100 个来源、~数百页的规模下工作得出奇地好"

### Fallback 机制

LLM 不可用时，自动回退到 `WikiRetriever`（规则匹配）：
- 5 维加权评分：关键词(0.4) + 分类(0.2) + 标签(0.2) + 关联(0.1) + 章节(0.1)
- 标题直接匹配额外加 0.5 分

## 知识库维护（Lint）

### `wiki_lint.py` — 四个操作

#### 1. 审计报告（`--report`）

```
📊 总览
  文章总数:           85
  雪纳瑞相关:         85 (100%)
  有交叉引用:         77 / 85
  降级文章:           2
  冲突文章:           7

📋 主题覆盖度（对比 target_topics.json）
  已覆盖: 48 / 52
  缺失主题:
    - 犬皮肤过敏与特应性皮炎
    - 紧急症状识别手册
```

#### 2. 清理（`--prune`）

把和雪纳瑞无关的文章移到 `wiki/_archive/`（可恢复）：

```python
SCHNAUZER_KEYWORDS = {"雪纳瑞", "schnauzer", "迷你雪纳瑞", ...}
def is_schnauzer_relevant(title, entry):
    text = (title + " " + " ".join(entry["keywords"])).lower()
    return any(kw in text for kw in SCHNAUZER_KEYWORDS)
```

实际效果：555 篇 → 85 篇，清理了 470 篇通用犬类疾病文章。

#### 3. 交叉引用（`--crossref`）

用 LLM 语义匹配为每篇文章选 3-5 个最相关的文章。分批处理（每批 20 篇），避免 LLM 输出截断。

优化前（关键词重叠）:
```
犬胰腺炎 → [index, 迷你雪纳瑞, 巨型雪纳瑞, 费用估算]  ❌ 全是无关的
```

优化后（LLM 语义）:
```
犬胰腺炎 → [高胆固醇症, 犬糖尿病, 犬肝分流, 犬胆囊阻塞]  ✅ 主题相关
```

#### 4. 修复降级（`--fix-degraded`）

降级文章是 build_wiki LLM 处理失败时的原文直接包装。修复方法：重置 `wiki_processed: false`，重新跑 build_wiki。

### 定期维护脚本

```bash
# crontab: 每周一凌晨 3 点
0 3 * * 1 /path/to/scripts/weekly_lint.sh
```

执行流程：审计报告 → 清理 → 交叉引用 → 重建索引 → 快速准确度测试。日志输出到 `data/logs/`。

## 遇到的问题和解决方案

### 问题 1: 爬虫不停止（PetMD 超过 max_pages 限制）

**现象**: `max_pages_per_source: 200` 配置了，但 PetMD 爬了 518 页还没停。

**根因**: 爬虫的搜索关键词 + 分类浏览 + 直接 URL 三种策略各自计数，没有共享全局计数器。

**解决**: 手动停止爬虫。518 页的 PetMD 内容已经足够。

### 问题 2: build_wiki 生成了 347 篇重复文章

**现象**: PetMD 的 500 篇英文文章每篇都变成一个独立的 Wiki 文件。"犬甲状腺功能减退" 和 "Hypothyroidism in Dogs" 变成两个不同的文件。

**根因**: `_normalize_topic_key()` 只做简单字符串小写化，不做语义匹配。

**解决**: 建立 `TOPIC_SYNONYMS` 同义词表（25 组中英文映射），分组时自动合并同主题文章。效果：从 566 分组降到 555 分组，核心主题（胰腺炎、膀胱结石、甲减等）的中英文文章被正确合并。

### 问题 3: index.md 229KB 导致 LLM 处理低效

**现象**: 把 555 个条目（每个带长摘要 + 章节列表）全塞进一个 index.md。

**解决**: 改为两层索引。顶层 index.md 从 229KB 缩到 **847 字节**。分类级索引最大的是 03-健康医疗，35KB。LLM 每步处理量大幅减少。

### 问题 4: 85% 的文章和雪纳瑞无关

**现象**: PetMD 爬虫抓了大量通用犬类疾病（沙皮热、拳师犬心肌病、柯利眼异常...），555 篇中只有 85 篇和雪纳瑞相关。

**解决**: `wiki_lint.py --prune` 用关键词判断相关性，把 470 篇无关文章移到 `_archive/`。保留 85 篇精准文章，检索准确度反而提升（减少干扰项）。

### 问题 5: 交叉引用质量很差

**现象**: 关键词重叠度方案导致每篇文章的 related 几乎一样（都关联到品种百科 + index），因为所有文章都有 "雪纳瑞" 这个关键词。

**解决**: 改用 LLM 语义匹配。把全部 85 篇标题+摘要发给 LLM，分批（每批 20 篇）让 LLM 选相关文章。胰腺炎终于关联到胰腺炎饮食管理而不是费用估算。LLM 不可用时自动回退到关键词匹配。

### 问题 6: 无法回答知识库覆盖不到的问题

**现象**: 用户问"帮我找训练教程"或"最新的雪纳瑞狗粮推荐"，系统只能复述知识库中的通用内容，无法给出具体的外部资源或最新信息。

**根因**: 系统只有知识库检索一条路径，没有联网能力。知识库是静态的，无法覆盖所有可能的问题。

**解决**: 集成 `web_search` tool：
- 用 DuckDuckGo HTML 搜索（无需 API key），返回前 5 条结果（标题 + 摘要 + URL）
- 注册为 LangChain tool，和 `memory_expand` 并列
- 更新 SYSTEM_PROMPT，增加第三类信息源说明，LLM 自主判断何时需要联网搜索
- **关键设计**：不是每次都搜索，而是 LLM 判断知识库不足时才调用（通过 tool calling 机制）

```
信息获取优先级：
  1. 记忆（用户偏好/对话摘要）→ 直接使用
  2. 知识库（Wiki 文章）→ 引用来源
  3. 网页搜索（web_search tool）→ 知识库不足时 LLM 主动调用
```

## 面试怎么讲

> **问: 你的知识库检索为什么不用向量数据库？**
>
> "我们参考了 Karpathy 的 LLM Wiki 方案。在百篇级别的知识库规模下，让 LLM 直接读一个精简的索引目录来判断相关性，比向量相似度检索效果更好。因为 LLM 能理解语义 — 用户说 '耳朵红红的' 它知道要找 '犬耳部感染'，但向量相似度可能匹配不上。我们做了两层索引：第一层是分类概览（~1KB），第二层是分类内的文章列表（~3KB），LLM 两步就能定位到相关文章。准确度测试 15 道题平均 9.6/10。"

> **问: 你的知识库是怎么从 7 篇扩展到 85 篇的？**
>
> "分两步：先跑爬虫从 PetMD 抓了 518 页权威来源，再用 LLM 为爬虫覆盖不到的主题（如手剥技术、中国养犬规定）生成 52 篇文章。然后用 build_wiki 的同义词分组把中英文同主题文章合并，最后用 lint 工具清理掉 470 篇和雪纳瑞无关的通用文章。最终保留 85 篇高质量、雪纳瑞聚焦的文章。"

> **问: 知识库怎么维护？**
>
> "我们有一个 wiki_lint.py 工具，实现了 Karpathy 提出的 lint 操作：审计报告（检查覆盖度、冲突、降级）、清理（移除无关文章）、交叉引用（LLM 语义匹配）、修复降级文章。做成了 weekly_lint.sh 脚本，可以挂 cron 每周执行。每次维护后跑准确度测试确保不退化。"

> **问: 知识库覆盖不到的问题怎么办？**
>
> "我们集成了 web_search tool。LLM 有三类信息源：记忆、知识库、网页搜索。当知识库中没有足够信息时（比如用户要求推荐具体产品或最新教程），LLM 会通过 tool calling 机制自动调用 DuckDuckGo 搜索，把搜索结果整合到回答中。这不是每次都搜，而是 LLM 自主判断需要时才搜 — 知识库能回答的就用知识库，回答不了的才联网。这样既保证了专业性（知识库内容经过审核），又有了兜底能力。"
